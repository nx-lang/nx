## Findings

### ✅ Verified - RF1 `NxWorkspaceDescriptorScope` leaks GCHandles when the constructor throws mid-loop
- **Severity:** Medium
- **Evidence:** `bindings/dotnet/src/NxLang.Runtime/Interop/NxWorkspaceDescriptorScope.cs:18-52`. The constructor pins identity/source byte arrays inside the per-module loop (`_identityHandles[index] = GCHandle.Alloc(...)` / `_sourceHandles[index] = GCHandle.Alloc(...)`), then validates the *next* module via `ArgumentNullException.ThrowIfNull(module)` and an empty-identity `ArgumentException` at the *top* of the subsequent iteration (lines 29-33). If validation fails for module N > 0, iterations 0..N-1 have already allocated pinned GCHandles, the loop throws, the descriptor-array handle at line 49 is never allocated, and no `Dispose` is called on the half-constructed scope — so those pinned handles leak for the lifetime of the process. The same pattern applies between `GCHandle.Alloc` of the identity handle on line 37 and the source handle on line 38 if the second allocation throws (e.g. under extreme memory pressure).
- **Recommendation:** Wrap the per-module loop in a `try`/`catch` that calls the existing `Dispose` cleanup path (or an equivalent private helper) before rethrowing, so partially-allocated handles are always released. Alternatively, move the argument validation (null / empty-identity checks) into a pre-pass that runs before any `GCHandle.Alloc` call.
- **Fix:** Added a validation pre-pass and wrapped descriptor/handle allocation in a `try`/`catch` that disposes partially allocated handles before rethrowing.
- **Verification:** Confirmed in `NxWorkspaceDescriptorScope.cs:18-54`. `ValidateWorkspace` at line 21 runs a pure validation pre-pass (null modules, empty identity) before any `GCHandle.Alloc` call. The per-module loop at lines 27-53 is wrapped in `try`/`catch` that calls `Dispose()` before rethrowing, and `Dispose` uses `IsAllocated` guards so half-initialized handle slots are safely freed. Addresses both the "validation of module N throws after N-1 handles are pinned" and "identity allocated but source allocation throws" edge cases.

### ✅ Verified - RF2 `NxWorkspaceDescriptorScope` copies source bytes, defeating the `ReadOnlyMemory<byte>` design
- **Severity:** Low
- **Evidence:** `bindings/dotnet/src/NxLang.Runtime/Interop/NxWorkspaceDescriptorScope.cs:36` — `byte[] sourceBytes = module.SourceUtf8.ToArray();`. The .NET spec (`specs/dotnet-binding/spec.md:3-18`) and design decision 1 describe `ReadOnlyMemory<byte>` specifically so hosts (Nexara draft rows) can pass source bytes from storage *without* an extra managed copy. `ToArray()` always allocates and copies, so every validate/build call re-copies every module's source bytes before pinning them for the native call.
- **Recommendation:** Use `module.SourceUtf8.Pin()` to obtain a `MemoryHandle` (or `MemoryMarshal.TryGetArray` when the underlying store is already an array, falling back to `ToArray()` otherwise). Keep the `MemoryHandle` alive for the scope and dispose it in `Dispose`. This preserves the zero-copy invariant the byte-oriented model was designed for.
- **Fix:** Replaced unconditional `ToArray()` with `MemoryMarshal.TryGetArray` so array-backed `ReadOnlyMemory<byte>` is pinned without copying, with a copy fallback for non-array-backed memory.
- **Verification:** Confirmed in `NxWorkspaceDescriptorScope.cs:73-92`. `PinSource` uses `MemoryMarshal.TryGetArray` to pin the underlying array segment in place (using `IntPtr.Add(..., segment.Offset)` to handle non-zero offsets), and only falls back to `ToArray()` when the `ReadOnlyMemory<byte>` is not array-backed. Zero-copy invariant preserved for the common case of byte-array-backed source payloads.

### ✅ Verified - RF3 Ambiguous workspace-to-library fallback resolves by path basename
- **Severity:** Low
- **Evidence:** `crates/nx-api/src/artifacts.rs:283-303` (`ProgramBuildContext::visible_library_by_logical_identity`) and the mirror at `:1828-1838` (`logical_library_by_identity`) both accept a match when `root_identity.ends_with(&format!("/{}", identity))`. A workspace module `app/main.nx` importing `"../ui"` normalizes to identity `ui`; any visible library whose canonical path ends with `/ui` (e.g. `/vendors/a/ui` *and* `/vendors/b/ui`) is considered equivalent, and the first sorted root wins. This is stricter than the design's "exact normalized identity match" rule stated in `design.md:110-111` / `module-imports/spec.md:3-25` and can silently bind a workspace import to the wrong tenant's library when two loaded roots share a basename.
- **Recommendation:** Either (a) require exact equality between the normalized library root identity and the import identity, or (b) when multiple roots match by suffix, report an ambiguous-import diagnostic naming the conflicting library roots instead of first-wins. Add a regression test with two libraries whose paths end in the same basename.
- **Fix:** Changed logical library fallback to prefer exact matches, report an ambiguity diagnostic when multiple suffix matches exist, and added a regression test with two visible `ui` libraries.
- **Verification:** Confirmed in `crates/nx-api/src/artifacts.rs:296-337` (primary resolver) and `:1893-1916` (mirror). `visible_library_by_logical_identity` now returns `LogicalLibraryResolution::Found` for a unique exact match, `Ambiguous` with the conflicting roots otherwise, and only falls back to suffix matching when no exact match exists. The mirror `logical_library_by_identity` also prefers exact match and returns `None` (treated as missing) on suffix ambiguity. The regression test `validate_workspace_reports_ambiguous_library_basename_fallback` (line 3304) loads two `ui` libraries under `tenant-a`/`tenant-b` and asserts the ambiguity diagnostic names both roots. Cargo test suite passes 84/84 lib tests in nx-api.

### ✅ Verified - RF4 Workspace `ProgramArtifact` does not carry the submitted source map for eval-time diagnostics
- **Severity:** Low
- **Evidence:** `crates/nx-api/src/artifacts.rs:59-66` (`ProgramArtifact` fields) and `crates/nx-api/src/eval.rs:93-108` (`program_root_source`). The source map built in `analyze_logical_module_graph` (`artifacts.rs:744-748`) is used for static-analysis diagnostics in `validate_workspace`/`build_workspace_program_artifact`, then dropped. At eval time, `program_root_source` only returns non-empty text for identities that `Path::new(...).is_file()` returns true for; for logical identities like `shared/value.nx` it returns `""`. The `no_root` diagnostic built at `eval.rs:122-129` then uses `full_source_span("")` → `TextRange(0, 0)` and an empty label source, yielding misleading line/column positions (1:1 → 1:1). The `artifact-model/spec.md` scenario "Workspace artifact preserves submitted modules separately" is satisfied by `root_modules`, but the source bytes themselves are not preserved, and the diagnostic-span-from-provider guarantee (`source-analysis-pipeline/spec.md:36-51`) silently degrades once the artifact is disconnected from the workspace.
- **Recommendation:** Either (a) store the submitted source-byte map (keyed by normalized identity) on `ProgramArtifact` and have `eval_program_artifact_with_source` consult it before falling back to filesystem reads, or (b) document that eval-time diagnostic spans for workspace-identity labels are not guaranteed and extend `eval_workspace_artifact_reports_no_root_for_selected_entry` (`artifacts.rs:3361`) to assert the actual span values produced.
- **Fix:** Added a source map to `ProgramArtifact`, updated eval/static diagnostic conversion to consult it before filesystem fallback, and extended the no-root workspace test to assert a non-empty source span.
- **Verification:** Confirmed in `crates/nx-api/src/artifacts.rs:78` (`pub(crate) source_map: FxHashMap<String, Arc<str>>`), populated from `analysis.source_map` in `build_program_artifact_from_graph` (line 1200, 1214). `program_root_source` (`eval.rs:94-111`) now consults `program.source_map` first and only falls back to filesystem when the identity is absent from the map. `program_artifact_error_diagnostics` (`eval.rs:90`) also passes the source map to `diagnostics_to_api_with_sources`. The regression test `eval_workspace_artifact_reports_no_root_for_selected_entry` (artifacts.rs:3566) now asserts `label.span.end_column > 1`, proving the source map provides real column positions rather than the prior empty-span fallback.

### ✅ Verified - RF6 Redundant UTF-8 validation of workspace source bytes
- **Severity:** Low
- **Evidence:** `crates/nx-ffi/src/lib.rs:288` validates `std::str::from_utf8(source_utf8)` for each descriptor while building the Rust `NxWorkspace`, and `crates/nx-api/src/source_graph.rs:116-120` re-runs `std::str::from_utf8` for the exact same bytes inside `WorkspaceSourceProvider::load_graph`. The first check is required at the FFI boundary to return `InvalidArgument` (per `workspace-programs/spec.md:117-120`); the second re-validates identical bytes the Rust-owned `NxWorkspaceModule` already produced. Functionally harmless, but every validate/build call pays the cost twice.
- **Recommendation:** Either (a) have the FFI skip the upfront UTF-8 check and rely on `WorkspaceSourceProvider` (converting the resulting error into `InvalidArgument` at the boundary), or (b) convert `WorkspaceSourceProvider` to skip the re-check when the `Vec<u8>` is known-valid. Option (a) keeps the single-source-of-truth in `nx-api`.
- **Fix:** Made `NxWorkspaceModule` construction validate UTF-8 once, store decoded source as `Arc<str>`, and keep module fields private so invalid source bytes cannot enter a workspace. FFI now validates borrowed source bytes as `&str` and copies directly into `Arc<str>` storage, while `WorkspaceSourceProvider` reuses the validated `Arc<str>` without another UTF-8 check.
- **Verification:** Confirmed in `crates/nx-api/src/workspace.rs:35-96`: `NxWorkspaceModule` stores `source: Arc<str>` behind private fields; `from_utf8` validates bytes once (line 71), `from_source` accepts already-decoded text without re-validating. `crates/nx-api/src/source_graph.rs:106-118` (`WorkspaceSourceProvider::load_graph`) now clones the validated `Arc<str>` via `module.source_arc()` with no UTF-8 check. The FFI at `crates/nx-ffi/src/lib.rs:287-292` calls `std::str::from_utf8` once and passes the resulting `&str` to `from_source`, which just materializes an `Arc<str>`. UTF-8 is validated exactly once per path.

### ✅ Verified - RF8 Workspace artifact preserves every analyzed module, even unreachable ones
- **Severity:** Low
- **Evidence:** `crates/nx-api/src/artifacts.rs:1125-1156` (`build_program_artifact_from_graph`) — every module the graph analyzed is pushed onto `root_modules` and into `ResolvedProgram::modules`. For a workspace of N modules where only k < N are reachable from the entry identity, the artifact still carries all N, their `LoweredModule`s, and their entries in `local_items`/`imports`. The spec at `artifact-model/spec.md:14-21` says "preserve separate `ModuleArtifact`s for the submitted workspace modules **that participate in analysis and execution**" — analysis covers all submitted modules, so this is arguably compliant, but it can inflate artifact memory footprint substantially for hosts (Nexara) that submit large workspaces and build many per-entry artifacts.
- **Recommendation:** Consider a reachability pass that trims `root_modules`/resolved-program contents to the transitive import closure of the entry identity. At minimum, document the inclusion rule on `ProgramArtifact::root_modules` so callers understand the memory footprint per build.
- **Fix:** Documented that `ProgramArtifact::root_modules` preserves every submitted module analyzed for the build, not only the selected entry module's transitive import closure.
- **Verification:** Confirmed at `crates/nx-api/src/artifacts.rs:61-65`: the `root_modules` doc-comment on `ProgramArtifact` now states "Workspace builds currently preserve every submitted module analyzed for the build, not only the transitive import closure of the selected entry module." Inclusion rule is documented; the trimming is left for a future change.

### ✅ Verified - RF9 `ResolvedModule.identity` mixes normalized logical identities and raw filesystem paths
- **Severity:** Low
- **Evidence:** `crates/nx-api/src/artifacts.rs:1530-1558` (`build_resolved_program`). For workspace/graph root modules, `artifact.file_name` is the normalized logical identity (e.g. `app/main.nx`). For library modules pushed by the subsequent loop (`:1544-1558`), `artifact.file_name` remains the raw canonical filesystem path produced by `read_library_source_files` (e.g. `/tmp/.tmpABC/ui/button.nx`). Both populate the same `ResolvedModule.identity` field, which `Interpreter::execute_resolved_program_module_function` at `crates/nx-interpreter/src/interpreter.rs:276` matches by string equality. The entry-module path is consistent (the entry identity is always a normalized logical identity and root modules store normalized identities), so the runtime dispatch is correct today, but mixing semantics on one field is a latent footgun — e.g. a future caller adding a library-entry feature would pass a raw path here.
- **Recommendation:** Either (a) normalize library module identities to logical ones during `build_resolved_program` for uniformity, or (b) split the field into `logical_identity` vs `source_file` so the semantic boundary is explicit. At minimum, add a doc comment on `ResolvedModule.identity` documenting that workspace/graph modules use normalized identities while library modules carry raw paths.
- **Fix:** Replaced the overloaded `ResolvedModule.identity` string with typed `ResolvedModuleSource` provenance. Source-provider modules now have a dedicated logical-identity index, library modules carry library root/module path provenance, `ProgramArtifact` records the selected entry `RuntimeModuleId` when available, and `eval_program_artifact` dispatches `root()` by runtime module id instead of string identity lookup. Added a regression test covering source-provider lookup versus library provenance.
- **Verification:** Confirmed across multiple files. `crates/nx-interpreter/src/resolved_program.rs:51-90` defines `ResolvedModule { id, source: ResolvedModuleSource, lowered_module }` with the enum distinguishing `SourceProvider { identity }` from `Library { root_path, module_path }`, plus dedicated `source_provider_modules` / `prepared_modules` indexes. `ProgramArtifact.entry_module_id: Option<RuntimeModuleId>` (artifacts.rs:69) is populated via `resolved_program.source_provider_module_id(entry_identity)` (artifacts.rs:1204). `eval.rs:146-166` dispatches `root()` by calling `interpreter.execute_resolved_program_module_function(entry_module_id, "root", vec![])` — no more string-identity lookup. The interpreter side (interpreter.rs:264-303) takes `RuntimeModuleId` directly. The regression test `build_workspace_program_artifact_records_typed_source_provenance` (artifacts.rs:3470-3526) constructs a workspace importing a library, asserts the entry module has `SourceProvider { identity == "app/main.nx" }` provenance, the library module has `Library { root_path, module_path }` matching the canonicalized library path, and that `module_by_source_provider_identity` does NOT resolve a library module's path — the semantic boundary is enforced by types, not by string coincidence.

## Questions
- **Q1 (Open Question from design.md):** The design's Open Questions list "Should a later change add extension fallback such as `./shared/questions` → `shared/questions.nx`?" The current implementation requires exact normalized identity match for workspace resolution, which matches the design's "exact normalized identity matches" decision — is that a firm no-for-now, or is a follow-up change already planned? If the former, an explicit test asserting that `import "../shared/questions"` does NOT resolve to workspace module `shared/questions.nx` would lock the contract in.
- **Q2 (Resolved by RF4 fix):** `program_root_source` now consults the artifact source map before disk fallback, so workspace identities that collide with on-disk filenames use submitted source text for eval-time diagnostics.
