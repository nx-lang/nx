## Findings

### ✅ Verified - RF1 Record-literal unknown-field diagnostics all point at the whole record-literal span
- **Severity:** Low
- **Evidence:** `crates/nx-types/src/infer.rs` (913-925) reports `unknown-record-field` diagnostics with `span` (the record literal's `span` parameter) for every unknown property. When a record literal supplies multiple unknown fields, every diagnostic ends up labeled at the same opening span instead of pointing at each individual key. By contrast, the element-style path uses each property's own span via `check_property_path_bindings` in `crates/nx-types/src/infer.rs` (1587-1627), so element-style errors locate the offending field.
- **Recommendation:** Carry per-field spans for record literals (e.g., extend `ast::Expr::RecordLiteral` to keep a span per `(Name, ExprId)` key, or fall back to the value expression's span via `module.raw_module().expr(*expr_id).span()` when a key span is unavailable) and pass that span to `self.error(...)`. As a smaller intermediate step, at least use the value expression's span so multi-error reports do not collapse onto a single source location.
- **Fix:** Added `RecordLiteralProperty` to carry property spans through HIR record literals and changed `infer_record_literal` to report `unknown-record-field` at each property's span, with regression coverage for multiple unknown fields.
- **Verification:** Confirmed `crates/nx-hir/src/ast/expr.rs` introduces `RecordLiteralProperty { name, value, span }`, `crates/nx-hir/src/lower.rs` populates `span` from the source `Property`, and `crates/nx-types/src/infer.rs` `infer_record_literal` now reports `unknown-record-field` with `property.span`. The new test `test_record_literal_unknown_record_fields_use_value_spans` asserts that two unknown fields produce two distinct diagnostic spans. RF6 below tracks the related `record-field-type-mismatch` span, which is not in scope of RF1's evidence.

### ✅ Verified - RF2 Assumed-package warning fires for dependency libraries even when no TypeScript import is actually emitted
- **Severity:** Low
- **Evidence:** `crates/nx-cli/src/codegen/languages/typescript.rs:99-122` (`collect_warnings`) iterates every `module.imported_types` entry and emits one warning per `(library_name, package_target)` pair regardless of whether the rendered module contains any reference to that dependency. The corresponding spec scenario `specs/cli-code-generation/spec.md` "TypeScript warns for assumed dependency package target" reads `WHEN TypeScript generation emits an import for dependency library '../question-flow'` and `AND the warning SHALL include the resolved package target that was emitted in generated source`, which ties the warning to actually emitting an import. With wildcard imports (e.g. `import "../question-flow"`) that the current module never references, the warning still fires even though no `import type` line is produced in the `.ts` output.
- **Recommendation:** Either (a) compute warnings during `render_module` from the imports actually written by `collect_module_imports`, recording the unique dependency package targets used, then surface those warnings up to the CLI, or (b) restrict `collect_warnings` to dependency packages that appear in `collect_module_imports` for at least one rendered module. Add a regression test where a library wildcard-imports a sibling library it never references and asserts no warning is produced.
- **Fix:** Restricted TypeScript assumed-package warnings to dependency package targets present in collected module imports, suppressed single-file TypeScript import warnings when imports are not emitted, and added a regression test for unused wildcard dependency imports.
- **Verification:** Confirmed `collect_warnings` now takes an `include_imports` flag, runs `collect_module_imports` per module, and only emits a warning when the assumed package target appears in the emitted imports map (`crates/nx-cli/src/codegen/languages/typescript.rs:99-135`). Single-file mode passes `include_imports=false` and produces no warnings. The new test `generates_no_typescript_dependency_warning_when_no_import_is_emitted` exercises an unused wildcard dependency and asserts the warnings list is empty.

### ✅ Verified - RF4 `infer_record_literal` looks up each field twice
- **Severity:** Low
- **Evidence:** `crates/nx-types/src/infer.rs` (913-941) iterates `properties` once to call `record_field_type_ref(...).is_none()` for the unknown-field diagnostic, then iterates `properties` again to call `record_field_type_ref(...)` and run `check_typed_binding`. Both loops perform the same linear scan over `effective_shape.fields` or `record_def.properties` per property. This is functionally correct but duplicates work for every record literal and forces the helper to be called twice per supplied field.
- **Recommendation:** Combine the two passes: in a single loop, call `record_field_type_ref` once and either emit `unknown-record-field` or run `check_typed_binding`. The branchless flow also makes it natural to fix RF1 in the same place because the loop already has the property's expression in scope.
- **Fix:** Combined record literal field validation into one loop so each supplied field calls `record_field_type_ref` once before either type-checking the value or reporting an unknown field.
- **Verification:** Confirmed `infer_record_literal` (`crates/nx-types/src/infer.rs:913-941`) now performs a single `match self.record_field_type_ref(...)` per property, with `Some(field_ty)` running `check_typed_binding` and `None` reporting `unknown-record-field`. The previous duplicate scan over `effective_shape.fields`/`record_def.properties` is gone.

### ✅ Verified - RF5 Cross-library type-alias references silently fall through without an emitted import
- **Severity:** Low
- **Evidence:** `crates/nx-cli/src/codegen/model.rs` (727-741) returns `None` from `CachedImportedLibrary::imported_type` for `PreparedItemKind::TypeAlias`, so a sibling library that exports a `type` alias is not added to `imported_types`. In `crates/nx-cli/src/codegen/languages/typescript.rs:574-612`, `add_imported_symbol` then takes the second path (`context.graph.owner_module(type_name)`), which only finds modules in the current library and returns early. The reference is left unimported, so generated TypeScript ends up with an unresolved bare name. This is pre-existing behavior in the imported-types model, but the new TypeScript dependency-import path in `add_imported_symbol` inherits the same limitation, and the spec scenarios talk about cross-library references in general (see `specs/cli-code-generation/spec.md` "TypeScript emits package imports for cross-library references"), without restricting them to `record`/`union`/`component` exports.
- **Recommendation:** Decide whether type aliases should participate in the cross-library import graph. If yes, extend `CachedImportedLibrary::imported_type` to accept `PreparedItemKind::TypeAlias` and adjust the C# emitter accordingly so the new TypeScript path automatically picks them up. If alias support is intentionally deferred, capture that as a `specs/future.md` entry and add a focused TypeScript test demonstrating the diagnostic message users will get today.
- **Fix:** Extended the imported-type model to include exported type aliases with resolved targets. TypeScript now imports cross-library aliases as type-only package symbols, while C# renders aliases transparently instead of emitting or referencing `global using` aliases.
- **Verification:** Confirmed `crates/nx-cli/src/codegen/model.rs` now produces `ImportedTypeKind::Alias { target, target_is_reference }` for `PreparedItemKind::TypeAlias` and resolves chained aliases via `resolve_alias_target`. TypeScript: `add_imported_symbol` reaches the dependency branch for visible aliased names and emits `import type { Exported as Local }` (covered by `generates_typescript_library_files_with_dependency_alias_imports`). C#: aliases inline transparently via `csharp_imported_alias_target_type` and the dependency-namespace warning is suppressed for primitive-target aliases (`generates_csharp_dependency_primitive_aliases_transparently`, `generates_csharp_dependency_nominal_aliases_as_target_types`). All 112 `nx-cli` tests pass. The C# behavior for same-library aliases now also bypasses `global using`; that is a deliberate consequence of the fix description but is broader than RF5's evidence and is captured as RF7 below.

## New Findings Discovered During 2026-04-27 23:56 Verification
### ✅ Verified - RF6 `record-field-type-mismatch` diagnostics still report at the outer record-literal span
- **Severity:** Low
- **Evidence:** After the RF1 fix, `crates/nx-types/src/infer.rs` `infer_record_literal` reports `unknown-record-field` at `property.span`, but the adjacent `Some(field_ty)` arm still calls `self.check_typed_binding(&actual, &expected, span, "record-field-type-mismatch", ...)` where `span` is the outer record literal span. A record literal that supplies multiple known fields with type mismatches will collapse all `record-field-type-mismatch` diagnostics onto the same source range, mirroring the original RF1 problem for the type-mismatch case. This is the exact pre-existing behavior RF1's recommendation flagged ("pass that span to `self.error(...)`"), but only the unknown-field branch was updated.
- **Recommendation:** In the `Some(field_ty)` branch of `infer_record_literal`, pass `property.span` to `check_typed_binding` (or carry the span through `check_typed_binding`) so per-field type-mismatch diagnostics also point at the offending value. Add a regression test analogous to `test_record_literal_unknown_record_fields_use_value_spans` for `record-field-type-mismatch`.
- **Fix:** Changed `infer_record_literal` to pass `property.span` to `check_typed_binding` for `record-field-type-mismatch` and added `test_record_literal_field_type_mismatches_use_value_spans` to assert multiple field mismatches report distinct property spans.
- **Verification:** Confirmed `crates/nx-types/src/infer.rs` now passes `property.span` to `check_typed_binding` in the known-field branch of `infer_record_literal`. The regression test `test_record_literal_field_type_mismatches_use_value_spans` asserts two `record-field-type-mismatch` diagnostics have distinct primary spans, and `cargo test -p nx-types test_record_literal_field_type_mismatches_use_value_spans --no-fail-fast` passes.

### ✅ Verified - RF7 Same-library C# `global using` aliases removed without spec or proposal coverage
- **Severity:** Low
- **Evidence:** The RF5 fix removes `emit_alias` entirely from `crates/nx-cli/src/codegen/languages/csharp.rs` and inlines local alias targets at every reference site (e.g., `csharp_type_name_inner` no longer overrides `alias_type.text` with the alias name). A module that exports only aliases now produces a header-only file with no `namespace` block; this is exercised by the renamed test `generates_csharp_alias_only_output_without_global_usings` and the new `generates_csharp_alias_references_transparently`. The proposal, design, and `specs/cli-code-generation/spec.md` only describe TypeScript cross-library imports and do not mention removing the long-standing C# `global using Foo = Bar;` output for same-library aliases.
- **Recommendation:** Decide whether the new "C# inlines all aliases" behavior is the intended public contract. If yes, add a `cli-code-generation` spec scenario that pins the new behavior (alias-only modules emit no C# output, alias references resolve to the target type, no `global using` is generated) so future regressions are caught. If no, restore `emit_alias` for same-library aliases while keeping the new cross-library alias inlining.
- **Fix:** Added the `C# aliases render transparently` scenario to `specs/cli-code-generation/spec.md`, pinning target-type rendering, no `global using`, and no namespace block solely for alias-only modules.
- **Verification:** Confirmed `openspec/changes/fix-authoring-codegen-safety-gaps/specs/cli-code-generation/spec.md` now includes the `C# aliases render transparently` scenario covering target-type rendering, no `global using`, and no namespace block for alias-only modules. The C# generator and tests match that contract, including `generates_csharp_alias_only_output_without_global_usings`, `generates_csharp_alias_references_transparently`, and cross-module/dependency alias coverage. `cargo test -p nx-cli --no-fail-fast` and `openspec validate fix-authoring-codegen-safety-gaps --strict` pass.

## Summary
The change implements the three target capabilities and the test coverage is strong: nullable-list widening is preserved against `T?[]` regressions, strict record-field validation runs at both the type-check and runtime layers, and TypeScript cross-library imports now flow through the existing `imported_types` model with explicit aliasing and an assumed-package warning. RF1–RF7 are verified, with no remaining review findings.
