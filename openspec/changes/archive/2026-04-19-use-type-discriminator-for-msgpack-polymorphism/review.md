## Findings

### ✅ Verified - RF1 No test coverage for nested or collection-shaped polymorphic records
- **Severity:** Medium
- **Evidence:** Task 3.1 requires "round-trip tests for raw->typed and typed->raw MessagePack polymorphic records, including nested/collection cases." The only new serialization tests are `PolymorphicTypedAction_SerializesAsMapWithTypeDiscriminator` and `PolymorphicTypedAction_RawToTypedRoundTrip_PreservesConcreteType` in `bindings/dotnet/tests/NxLang.Runtime.Tests/NxRuntimeComponentTests.cs:299-329`; both only exercise a single top-level `SearchAction`. No test covers `List<SearchAction>`, `SearchAction[]`, or a record containing a polymorphic property. The formatter's deserialize path roundtrips through `Dictionary<string, object?>` then re-serializes each property value before deserializing it to its declared type (`NxPolymorphicMessagePackFormatter.cs:70-108`); that path is where nested polymorphism is most likely to silently break (type coercion of boxed numerics, nested maps re-dispatched through the outer resolver, polymorphic elements of a generic list). Without tests, regressions here will be invisible.
- **Recommendation:** Add at least two tests: (a) a record whose field is a polymorphic `SearchAction`, round-tripped raw↔typed; (b) a `SearchAction[]` (or `List<SearchAction>`) round-tripped raw↔typed. Include a multi-level case using the `Question → TextQuestion → ShortTextQuestion` shape from the specs.
- **Fix:** Added `PolymorphicTypedPayload_RoundTripsNestedAndCollectionValues` in `bindings/dotnet/tests/NxLang.Runtime.Tests/NxRuntimeComponentTests.cs` to cover nested polymorphic fields (`SearchAction`, `SearchAction[]`, and `Question` hierarchy) and assert concrete-type preservation.
- **Verification:** Confirmed tests `PolymorphicTypedPayload_RoundTripsNestedAndCollectionValues` and `PolymorphicTypedPayload_RoundTripsNestedPayloadsWhenDiscriminatorIsNotFirst` exist and pass under `dotnet test bindings/dotnet/NxLang.sln --filter "FullyQualifiedName~NxRuntimeComponentTests.PolymorphicTyped"`.

### ✅ Verified - RF2 Deserialize path roundtrips every field through an intermediate MessagePack buffer
- **Severity:** Medium
- **Evidence:** `NxPolymorphicMessagePackFormatter.Deserialize` first decodes the whole payload into `Dictionary<string, object?>`, then for each declared property does `MessagePackSerializer.Serialize(rawValue, options)` followed by `MessagePackSerializer.Deserialize(propertyType, bytes, options)` (`NxPolymorphicMessagePackFormatter.cs:70-108`). That is O(N) buffer allocations per record and loses the direct binary stream. It also introduces a subtle semantic coupling: any formatter that depends on sequence-level context (e.g., extension types, raw bytes) is re-run on synthesized bytes instead of the original stream. For NX's current NxValue shapes this is likely tolerable, but it's a footgun as the type system grows.
- **Recommendation:** Switch to a single-pass read. Either: (1) read the map length, then for each of the N entries read the key string and dispatch on it — writing directly into the instance via the cached `PropertyInfo`; or (2) decode into a buffered reader, locate `$type`, instantiate, then re-scan keys with typed per-key dispatch. Both avoid the intermediate `Dictionary<string, object?>` and its allocations.
- **Verification:** `Deserialize` in `NxPolymorphicMessagePackFormatter.cs` now uses `ReadMapHeader`, `PeekDiscriminator` on `reader.CreatePeekReader()` to locate `$type` without advancing the main reader, instantiates the concrete type, then dispatches each map entry with `MessagePackSerializer.Deserialize(propertyType, ref reader, options)` (no `Dictionary<string, object?>` and no per-field `Serialize` roundtrip). Covered by polymorphic formatter tests including discriminator ordering and unknown keys.

### ✅ Verified - RF3 MessagePack polymorphism registry is built from `JsonDerivedTypeAttribute`
- **Severity:** Medium
- **Evidence:** `NxPolymorphicMessagePackFormatter.BuildDiscriminatorMap` iterates `JsonDerivedTypeAttribute` on `TBase` to resolve `$type` → CLR type (`NxPolymorphicMessagePackFormatter.cs:114-128`). For generated DTOs this always works because the generator always emits `[JsonDerivedType(...)]` alongside `[MessagePackFormatter(...)]`. But as a managed public type (`[CLSCompliant(false)] public sealed class NxPolymorphicMessagePackFormatter<TBase>`), it's usable by hosts directly; a hand-written polymorphic root that only targets MessagePack would compile fine and then fail at runtime with "No $type discriminator registration" because it lacks the JSON attributes. Also, `JsonDerivedTypeAttribute.TypeDiscriminator` is typed `object?`; the filter `discriminatorValue is not string discriminator` silently drops integer discriminators rather than erroring.
- **Recommendation:** Either (a) mark the formatter `internal` so it's purely a code-generator implementation detail, (b) add a dedicated NX-owned attribute (e.g., `NxPolymorphicTypeAttribute`) that the generator emits in addition to `JsonDerivedTypeAttribute`, or (c) document that the formatter depends on `[JsonDerivedType]` with string discriminators and assert/throw a clearer error when the registry is empty at static-ctor time.
- **Fix:** Added formatter remarks documenting the `JsonDerivedTypeAttribute` dependency, plus explicit startup exceptions for non-string discriminators and empty discriminator maps in `NxPolymorphicMessagePackFormatter<TBase>`.
- **Verification:** Confirmed `<remarks>` on the formatter plus `BuildDiscriminatorMap` throws `InvalidOperationException` for non-string discriminators and for an empty map after attribute scan.

### ✅ Verified - RF4 Raw-to-typed test does not use real runtime-produced bytes
- **Severity:** Low
- **Evidence:** `PolymorphicTypedAction_RawToTypedRoundTrip_PreservesConcreteType` (`NxRuntimeComponentTests.cs:315-329`) synthesizes the "raw" payload from a `Dictionary<string, object?>` rather than using bytes produced by the NX runtime. The dotnet-binding spec scenario ("Typed MessagePack polymorphic record deserialization accepts canonical `$type` map values") is about bytes "produced from canonical raw runtime output". The synthesized dictionary does not prove parity with the native runtime's actual encoding — especially around key ordering, string interning, and numeric widths — which is the specific drift the change is meant to eliminate.
- **Recommendation:** Add a test that evaluates a small NX program producing a polymorphic record (e.g., `SearchRequested { query: "docs" }`) with MessagePack output, then deserializes those exact bytes through the typed `SearchAction` formatter and asserts concrete-type/field preservation.
- **Fix:** Added `PolymorphicTypedAction_DeserializesRuntimeProducedCanonicalBytes` to deserialize `SearchAction` from bytes returned by `NxRuntime.EvaluateBytes(...)` on an NX program emitting a polymorphic action value.
- **Verification:** Confirmed test calls `NxRuntime.EvaluateBytes` and deserializes into `SearchAction`; passes after correcting the embedded NX program to use a valid action element literal (see NF1). Same filter run as RF1.

### ✅ Verified - RF5 Generator emits `[MessagePackObject]` on abstract polymorphic roots alongside `[MessagePackFormatter]`
- **Severity:** Low
- **Evidence:** `emit_record` in `crates/nx-cli/src/codegen/languages/csharp.rs:243-286` always writes `[MessagePackObject]` before the class header, even for abstract records that also carry `[MessagePackFormatter(typeof(NxPolymorphicMessagePackFormatter<T>))]`. Two attributes nominally describing serialization behavior coexist on the same type. The custom formatter wins via resolver precedence, but the combination is noisy, can confuse readers who expect `[MessagePackObject]` to imply key-based member serialization, and is a mild foot-gun if MessagePack's source generator is ever enabled in the project (it may attempt to code-gen a formatter for the abstract class).
- **Recommendation:** Skip emitting `[MessagePackObject]` on abstract records that already have `[MessagePackFormatter(...)]`; or conversely drop the custom `[MessagePackFormatter]` from the abstract class and register the formatter via a resolver. Either choice removes the ambiguity.
- **Fix:** Updated C# codegen to skip emitting `[MessagePackObject]` on abstract roots that emit polymorphic formatter metadata (`[MessagePackFormatter(typeof(NxPolymorphicMessagePackFormatter<T>))]`).
- **Verification:** Confirmed `emit_record` in `csharp.rs` wraps `[MessagePackObject]` in `if !record.is_abstract || !should_emit_json_polymorphism_attributes(record, context)`.

### ✅ Verified - RF9 `ResolveWireName` converts int `Key` to a string
- **Severity:** Low
- **Evidence:** `NxPolymorphicMessagePackFormatter.ResolveWireName` returns `key.IntKey.Value.ToString()` for int-keyed properties and the serializer then writes that as a string map key (`NxPolymorphicMessagePackFormatter.cs:158-167`, `Serialize` at `:50-59`). For any type using `[Key(0)]`, `[Key(1)]`, ... this encodes the key as the string `"0"`, `"1"`, etc., rather than as an integer — both diverging from normal MessagePack-contract semantics and from any canonical NxValue that might use int keys. The generator always emits string `[Key("name")]`, so this is not a live bug, but the fallback silently corrupts int keys instead of rejecting them.
- **Recommendation:** Either (a) throw if `IntKey` is present (the formatter is NX-specific and NX uses string wire names), or (b) emit an int map key via a separate code path. Option (a) is simpler and safer given the stated contract.
- **Fix:** Updated `ResolveWireName` to throw a clear `InvalidOperationException` when an integer MessagePack key is encountered instead of coercing it into a string key.
- **Verification:** Confirmed `ResolveWireName` throws when `key.IntKey` is set in `NxPolymorphicMessagePackFormatter.cs`.

### ✅ Verified - RF10 Formatter's typed-to-dict coercion can corrupt nested numeric types
- **Severity:** Low
- **Evidence:** `Deserialize` reads the outer map as `Dictionary<string, object?>`, which decodes MessagePack integers into the smallest fitting CLR numeric type (byte/short/int/long/ulong). Re-serializing via `MessagePackSerializer.Serialize(rawValue, options)` then re-deserializing to the property type via `MessagePackSerializer.Deserialize(propertyType, bytes, options)` is generally tolerant, but there are edge cases: a property declared `long` that receives a positive value encoded as a negative-looking `sbyte` through `object`-typed intermediate, or a property declared `decimal`/`BigInteger` (non-NX today but plausible as the DTO surface grows). Combined with RF2, this is an argument for eliminating the intermediate dictionary entirely.
- **Recommendation:** Handled by the same fix as RF2 (single-pass key-dispatched deserialization bypasses this risk).
- **Verification:** Same as RF2: deserialize path no longer boxes map values as `object?` for declared properties; numeric coercion risk from that pattern is removed for the polymorphic formatter's outer map.

## Questions

The answer determines whether the JsonDerivedType coupling (RF3) is acceptable or whether the class should move to an internal namespace.

## Summary
Implementation matches the specs at the attribute/shape level and deletes the `[Union]` path cleanly. Formatter deserialize no longer uses a `Dictionary<string, object?>` plus per-field buffer re-encode; it streams map entries and deserializes declared properties in place, with tests for discriminator position, unknown keys, and nested payloads. Remaining gap called out in this review is documentation depth for migration/wire-shape (RF8). Verification run: `dotnet test bindings/dotnet/NxLang.sln --filter "FullyQualifiedName~NxRuntimeComponentTests.PolymorphicTyped"` (10 passed); `cargo test -p nx-cli generate_types_warns_when_csharp_abstract_root_has_no_concrete_descendants`; `cargo test -p nx-cli polymorphism`; `dotnet test` filter `DispatchComponentActions_WithPersistedStateSnapshot_SucceedsAcrossCalls` after codegen/formatter fix for RF11.
